PyperCache Reconstruction Blueprint

One document for rebuilding the entire repository

This page collapses the README, module structure, tests, examples, and implementation details into one reconstruction-grade reference. If a developer understands the contracts documented here, they should be able to recreate the system in this repo with the same behavior and tradeoffs.

What PyperCache is

A durable, file-backed cache for JSON-like payloads with pluggable persistence backends, optional TTL, typed object hydration, read-only in-memory query navigation, append-only request logging, and a sync-first HTTP wrapper layered on top.

What it is not

It is not a relational database, not a cross-record query engine, not a schema-enforced document store, and not a distributed cache. The center of gravity is key -> payload storage plus convenient hydration and retrieval.

Primary design rule

Keep the public API small while pushing complexity into a few focused subsystems: storage, typing/model hydration, query navigation, and HTTP-wrapper ergonomics.

North Star

The repo is easiest to reconstruct if you think in contracts rather than files. PyperCache has five user-facing promises:

  1. A cache key maps to one stored record containing payload, timestamp, expiry, and optional cast metadata.
  2. The storage backend is selected only from the cache filepath extension, while cache semantics stay stable.
  3. Stored payloads can be re-hydrated into Python objects through a lightweight class registry and type instantiation helper.
  4. Loaded payloads can be navigated in memory through a small selector language without touching storage.
  5. API clients can compose requests, caching, logging, response decoding, and type hydration behind one sync wrapper.
Rebuild target: Match behavioral semantics before matching implementation details. The tests in this repo lock down many edge cases, especially around missing values, stale data, lazy model fields, and SQLite write-behind behavior.

Core Contracts

These are the persistent and in-memory data shapes a reimplementation must preserve.

CacheRecord dict

{
  "cast": "builtins-name or module.Class",
  "expiry": 300 or "math.inf",
  "timestamp": 1713800000.123,
  "data": { ... any payload ... }
}

"math.inf" is the wire format for infinite TTL because JSON cannot encode math.inf safely.

Request log line

{
  "uri": "https://service/path?x=1",
  "status": 200,
  "timestamp": 1713800000.123
}

Each log write is one JSON object per line. The logger reads JSONL only and does not migrate older array-format files.

Chunk manifest

{
  "chunk_size_in_bytes": 15728640,
  "chunks_path": "path/to/directory",
  "chunks_count": 3,
  "chunks_map": {
    "key-a": "0-chunk.pkl",
    "key-b": "1-chunk.pkl"
  }
}

Chunk files themselves are pickle payloads that store small dict fragments.

SQLite row schema

cache_records(
  key TEXT PRIMARY KEY,
  cast TEXT,
  expiry REAL,
  timestamp REAL,
  data BLOB
)

The row is columnar. Payload bytes use a JSON -> msgpack -> jsonpickle fallback ladder.

Cache Layer

pypercache.core.cache.Cache is the central facade. It owns almost no persistence logic itself. Its job is to choose a storage backend, normalize record metadata, and expose a stable API.

Constructor behavior

  • If no filepath is passed, use api-cache.pkl.
  • Choose the backend strictly from the filepath suffix.
  • Create a per-instance ClassRepository() handle, even though the repository itself is a singleton.

Main methods

  • has(key): existence only.
  • is_data_fresh(key): existence plus non-stale TTL.
  • get(key): returns a CacheRecord or raises KeyError.
  • get_object(key): hydrates stored payload using the record's cast metadata.
  • store(), update(), completely_erase_cache(), close().

Store path

Cache.store(key, data, expiry=math.inf, cast=None)
  -> convert expiry to "math.inf" if needed
  -> convert cast to stored string
     - builtin types use short names like "dict"
     - user types use fully-qualified names like "module.Widget"
  -> create raw record dict with current timestamp
  -> hand off to storage.store_record()

Retrieval path

Cache.get(key)
  -> storage.get_record(key)
  -> CacheRecord(raw_record_dict)

Cache.get_object(key)
  -> read CacheRecord
  -> require a valid cast type or raise AttributeError
  -> instantiate_type(record.cast, record.data)
Important semantic split: get() returns the wrapper record, while get_object() returns a hydrated Python object. Missing keys can use a default only in get_object(); missing get() always raises.

CacheRecord behavior

  • Lazily resolves the stored cast string into a class through look_up_class().
  • Exposes .query, which lazily builds and caches a JsonInjester over .data.
  • update(data) replaces the payload, refreshes the timestamp, and invalidates the cached injester.
  • is_data_stale compares time.time() against timestamp + expiry.
  • __repr__ encodes cast label plus data_fresh or data_stale.

Class registration

PyperCache has two ways to make stored payloads re-hydratable. @Cache.cached registers a class in the global repository. @apimodel also registers its decorated class, so both the simple and advanced model systems feed the same retrieval path.

Storage Layer

Storage is intentionally abstracted behind StorageMechanism. The common contract is small: load, save, get a record, update a record, store a record, erase everything, and touch the store on disk.

Extension Backend Persistence model When it fits
.pkl PickleStorage Whole-dataset pickle read/write Default, simple, opaque, fast enough for many cases
.json JSONStorage Whole-dataset JSON or jsonpickle Human-readable files or lightweight inspection
.manifest ChunkedStorage Chunk manifest plus per-chunk pickle files Large caches where per-key writes matter more than one-file simplicity
.db SQLiteStorage Columnar SQLite table with RAM mirror and write-behind batching High read volume or many writes where fsync cost matters

StorageMechanism base class

  • Owns a thread lock and the live records mapping.
  • Calls ensure_dirs_exist() and touch_store() before load/save.
  • Treats records as MutableMapping[str, dict], not necessarily a plain dict.
  • Returns CacheRecord wrappers from raw record dicts via get_record().

JSONStorage and PickleStorage

These are straightforward whole-file backends. Update means: build a CacheRecord from the raw dict, call record.update(data), then flush the entire dataset back to disk.

  • JSONStorage tries json.dumps() first and falls back to jsonpickle for complex values.
  • PickleStorage delegates file IO to PickleStore.

ChunkedStorage and ChunkedDictionary

This backend spreads keys across multiple pickle chunk files and keeps a JSON manifest mapping keys to chunk filenames. The chunk dictionary behaves like a dict but only loads chunk files on demand. 

chunks.manifest
  -> key "user:1" maps to "0-chunk.pkl"
  -> key "user:999" maps to "2-chunk.pkl"

0-chunk.pkl -> {"user:1": {...}, "user:2": {...}}
2-chunk.pkl -> {"user:999": {...}}
  • New keys go into the last chunk until the estimated JSON byte size would exceed the chunk limit.
  • Updating an existing key edits only its owning chunk and then re-saves that chunk plus the manifest.
  • erase_everything() deletes all chunk pickle files and clears the manifest state.

SQLiteStorage

This is the most opinionated subsystem in the repo. Its design goal is simple: keep reads RAM-fast and collapse many writes into a single SQLite transaction. 

open store
  -> open sqlite connection
  -> enable WAL, NORMAL sync, 8 KB pages, 64 MB cache
  -> bulk load every row into _BufferedMapping
  -> start background flush thread

store_record()
  -> mutate _BufferedMapping only
  -> mark key dirty
  -> flush immediately only if dirty threshold reached

flush()
  -> pop dirty upserts + deletes
  -> BEGIN
  -> executemany(UPSERT)
  -> executemany(DELETE)
  -> COMMIT

Serialization ladder for SQLite BLOBs

  1. None stays NULL.
  2. bytes and bytearray stay raw bytes.
  3. JSON-able values are stored as UTF-8 JSON bytes.
  4. Values that JSON cannot encode but msgpack can are stored as b"\x00" + msgpack.
  5. Everything else falls back to b"\x01" + jsonpickle JSON.

Durability tradeoff

Writes live in memory until a flush. A crash between flushes can lose up to the current dirty buffer plus roughly FLUSH_INTERVAL_SECONDS of changes. The repo treats that as acceptable because the system is a cache, not a source of truth.

Close behavior matters: SQLiteStorage.close() sets the closed flag, performs a final flush, checkpoints WAL with TRUNCATE, and closes the connection. Any serious reimplementation should keep this.

Query Layer

JsonInjester is a read-only selector engine over a single loaded JSON-like payload. It never talks to the storage backend. It only walks already-loaded dicts and lists.

Selector grammar

path                meta.total
match filter        users?role=admin
numeric match       items?score=#10
exists filter       users?role
pluck               users?name*
tail after filter   users?role=admin.address.city
quoted keys         "content-type"."x-rate-limit"

Execution model

  • Every get() call parses the selector with Lark into small AST objects.
  • The cursor starts at the root or the optional root= subpath.
  • Plain path segments walk dicts only.
  • Filters and plucks can turn a dict cursor into a list cursor or vice versa.
  • default_tail is applied only when the final result is a dict.

Missing-value rules

  • Missing path pieces yield the shared UNSET sentinel.
  • If default_value is provided, missing or None results return that default.
  • Falsy but real values like False, 0, and "" pass through unchanged.
  • select_first=True unwraps the first list element or returns UNSET for an empty list.

Non-obvious rules from tests

  • Integer list indexing like users.0.name is not supported.
  • A bare path against a list root raises TypeError; callers must use pluck or filter syntax.
  • Quoted keys are required for hyphenated names.
  • Unquoted non-ASCII keys are rejected by the grammar.
  • On dict-of-dicts filters, matches are returned as (key, value) tuples.
  • cast= is applied only when the final result is a dict.
q = cache.get("search:v1").query
q.get("hits?role=staff.name")                # ["Alice", "Carol"]
q.get("hits?role=staff", select_first=True)  # {"name": "Alice", ...}
q.get("meta.total", default_value=0)         # 3
q.has("meta.cursor")                         # False

Model Layer

The model subsystem is the most compositional part of the codebase. It combines a singleton class registry, a generic type instantiator, annotation metadata, lazy descriptors, and optional validation.

ClassRepository and instantiate_type

  • ClassRepository is a singleton mapping short class names and fully-qualified names to types.
  • look_up_class() resolves builtin primitive names first, then the repository, then import-by-path fallback.
  • instantiate_type() supports Annotated, list[T], dict[K, V], dataclasses, from_dict(), and best-effort constructor calls.

@apimodel decoration pipeline

@apimodel(validate=False, strict=False)
class Widget:
    id: int
    profile: Lazy[Profile]
    created_at: Annotated[datetime, Timestamp()]

decoration time:
  -> register class in ClassRepository
  -> resolve annotations with get_type_hints(include_extras=True)
  -> split fields into eager and lazy buckets
  -> install one LazyDescriptor per Lazy field
  -> generate __init__, __setattr__, as_dict, from_dict, __repr__, __eq__

Alias

Reads and writes a field through an alternate raw payload key. Supports dotted raw paths, so assigning display_name can write into profile.name.

Timestamp

Parses ISO strings, numeric strings, or numeric Unix timestamps into datetime. It can also serialize assigned datetimes back into raw payload shape using ISO or a supplied format string.

Columns

Converts a dict-of-arrays payload into list[RowModel]. On assignment, row objects or row dicts are columnized back into dict-of-arrays form.

Lazy

Marks a field for deferred hydration. The descriptor reads from the raw dict on first access, instantiates once, caches per instance, and can be cleared with del obj.field.

Validation semantics

  • strict=True means missing fields raise ApiModelValidationError instead of storing UNSET.
  • validate=True means hydrated values must match the annotation after casting.
  • UNSET is accepted by type validation unless strict=True requires presence.
  • Lazy fields are special: under validate=True or strict=True, they are checked during __init__ and cached immediately.

Mutation semantics

  • as_dict() returns the original raw dict object, not a copy.
  • Assigning to eager or lazy fields writes back into the raw dict using alias/timestamp/columns rules.
  • Assigning an already-instantiated apimodel instance preserves the instance and writes its raw dict shape.
  • Only a class's own annotations are considered; inherited annotations are intentionally ignored.
@apimodel(validate=True)
class HourlyPoint:
    time: Annotated[datetime, Timestamp(unit="seconds")]
    temperature_c: Annotated[float, Alias("temperature_2m")]
    weather_code: int

@apimodel(validate=True)
class Forecast:
    hourly: Lazy[
        Annotated[
            list[HourlyPoint],
            Columns(required=("time", "temperature_2m"))
        ]
    ]

API Wrapper

ApiWrapper is a sync-first base class for small HTTP clients. It is not a huge framework. It exists to standardize request composition, cache keys, type hydration, download helpers, and SSE parsing.

Constructor contract

  • origins must be non-empty.
  • default_origin must be one of the origin keys.
  • If cache_path is supplied, create a Cache; otherwise caching is disabled.
  • If request_log_path is supplied, create a RequestLogger.
  • If no session is supplied, the wrapper owns its own requests.Session.

Request pipeline

request(method, path, ...)
  -> normalize method and selected origin
  -> build final URL
  -> drop None values from params/data
  -> convert dataclasses and model-like inputs to JSON-safe values
  -> compute deterministic cache key:
       "api:{origin}:{sha256(canonical payload)}"
  -> for GET + json/auto responses:
       serve fresh cached value if present
  -> otherwise perform HTTP request
  -> write request log entry if enabled
  -> raise ApiHTTPError for 4xx/5xx with parsed body
  -> decode response to json/text/bytes/None
  -> cache only JSON-like GET responses
  -> return casted result if cast= was supplied

Decoding rules

  • expected="bytes" returns raw bytes.
  • expected="text" returns text.
  • expected="json" forces JSON parsing.
  • expected="auto" inspects content type and body emptiness.
  • An empty body returns None.

Caching rules

  • Only GET requests are cache candidates.
  • Only expected in {"auto", "json"} can be cached.
  • Only JSON-serializable result shapes are stored.
  • Cached payloads are wrapped as {"value": result}.

Other wrapper features

  • download_to() fetches bytes and writes them to a destination path.
  • stream_sse() parses server-sent events into SSEEvent dataclass instances.
  • _file_tuple() prepares upload tuples and request() always closes opened file handles.
  • close() flushes cache storage if needed and closes only sessions the wrapper created itself.
Origin handling: relative paths are joined against the chosen origin, while absolute URLs bypass origin lookup. Because origin is part of the cache-key payload, identical paths on different hosts do not collide.

Request Logging

RequestLogger is deliberately independent from the cache. It is an append-only audit trail, not a cache index and not a generalized analytics system.

  • Writes are protected by a thread lock.
  • log(uri, status) appends one JSON line and updates the in-memory record list.
  • get_logs_from_last_seconds(n) filters by timestamp and returns oldest-first.
  • The constructor touches the target file and loads all existing records into memory.
  • The logger reads JSONL only and does not rewrite older array-format files.
logger = RequestLogger("api_requests.log")
logger.log("/v1/items", 200)
recent = logger.get_logs_from_last_seconds(60)

Examples

The two example projects show the library at two different maturity levels.

Open-Meteo example

  • Best demonstration of the final architecture.
  • Uses ApiWrapper with named origins for geocoding and forecast hosts.
  • Uses @apimodel(validate=True) heavily.
  • Uses Alias, Timestamp, Columns, and Lazy together.
  • Shows how to turn awkward upstream API payloads into a clean object graph.

PokeAPI example

  • Instructional example of the older manual style.
  • Uses @Cache.cached dataclasses plus cast= on store.
  • Implements a soft rate limiter outside the core library.
  • Implements background refresh by keeping a registry of cache keys to fetch functions.
  • Good for understanding the minimal ingredients before adopting ApiWrapper.
class OpenMeteoClient(ApiWrapper):
    def search(...):
        return self.request(
            "GET",
            "/search",
            expected="json",
            cast=GeocodingResults,
            expiry=86400,
            origin="geocode",
        )

Test Matrix

The repo's tests are not just sanity checks. They define the behavior a replacement implementation must reproduce. If two interpretations of the code disagree, the tests win.

Test file What it locks down
test_cache.py Core cache CRUD, TTL freshness, cast retrieval, persistence, and backend dispatch.
test_cache_record.py Record serialization, class lookup, staleness semantics, and lazy cast resolution.
test_storage.py Factory mapping, storage contract, and chunked dictionary behavior.
test_sqlite_storage.py SQLite correctness plus explicit performance thresholds for writes, reads, flushes, and reopen times.
test_json_injester.py Selector grammar semantics, defaults, quoting, list-root errors, plucks, filters, and type casting.
test_apimodel.py Decoration-time behavior, lazy fields, aliasing, timestamp parsing, assignment semantics, and validation edge cases.
test_apimodel_columns.py Dict-of-arrays hydration, row-to-column assignment round trips, and timestamp handling on lists.
test_api_wrapper.py Caching policy, hashed cache keys, origin handling, decoding, SSE parsing, logging, and session ownership.
test_request_logger.py JSONL format, handling of non-JSONL files, time-window filtering, and thread-safe writes.

Performance contracts worth keeping

  • 10,000 SQLite in-memory writes should stay under 500 ms.
  • 10,000 SQLite reads should stay under 50 ms.
  • Flushing 1,000 rows should stay under 500 ms.
  • Flushing 10,000 rows should stay under 2 seconds.
  • Batch flush should be at least 5x faster than naive per-write commits.

Edge cases people usually miss

  • False, 0, and empty strings are real values, not missing values.
  • Deleting a lazy field cache must force re-hydration on next access.
  • Alias writes can target nested raw paths.
  • validate=True can pre-cache lazy values during construction.
  • SQLite delete-before-flush must not leave ghost rows on disk.

Rebuild Order

If you had to recreate the repository from scratch, do it in this order. Each step gives the next one stable dependencies and keeps the surface area under control.

  1. Implement the utility layer: UNSET, filesystem helpers, PickleStore, ClassRepository, and instantiate_type().
  2. Implement CacheRecord with TTL logic, lazy cast resolution, and query support.
  3. Implement StorageMechanism, then PickleStorage and JSONStorage, then the extension-based factory.
  4. Implement JsonInjester and lock its semantics against the query tests before building more on top.
  5. Implement Cache and make sure get() and get_object() split correctly.
  6. Implement RequestLogger as a simple JSONL append-only logger.
  7. Implement ChunkedDictionary and ChunkedStorage.
  8. Implement the model stack: field metadata, validation, lazy descriptor, and @apimodel.
  9. Implement SQLiteStorage last, after the record contract is already stable.
  10. Implement ApiWrapper once cache, typing, and logging are trustworthy.
  11. Recreate the examples to prove the abstractions are usable, not just technically correct.
  12. Run the full test suite and treat failures as contract mismatches, not optional cleanup.
Definition of done: a developer can move from raw API payload -> cache record -> typed object -> read-only selector query -> origin-aware HTTP wrapper without hidden behavior gaps between backends.
